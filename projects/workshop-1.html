Step 1: Git and GitHub - Forking

What are we doing?

Confused by all these tools and acronyms? Here's a quick rundown!

What is git?

Git is a tool software engineers use to better manage their code. With large projects where there may be hundreds or even thousands of files, engineers will need to have several "versions" of their code. For example, you may have a project that needs a production version of code (for code that is being ran "in production," or live for customers), a development version, or a new-feature version. Git allows people to easily switch between version and contribute to changing projects.

Some benefits of git include:

  • Many people can work on the same codebase at the same time
  • Code can be "version controlled" (e.g. version 2.0, 2.1, 2.2.4, etc.)
  • You can "undo" or "revert" back to old/working version of your code very easily.

For every project you have, you'll have one git repository. A repository (or "repo") is basically like a folder, or directory. You can have as many files or folders inside this repo as you wish.

What is GitHub?

GitHub is a website that displays git repositories on an open and public website. You're able to find projects and code that other people have written, share your work with others, and contribute to other people's projects.

What is Forking?

Forking on GitHub means you are making a copy of someone else's repo for your own use. Usually, you do this when you want to contribute back to a repository.

For example, Facebook has a few open source projects on GitHub, and their most popular repo is React. React is a JavaScript library that builds user interfaces for web projects.

Since React is open source, anyone can go in and check out the source code for React. Anyone could "contribute" to, or help, the React Project.

However, only a select few people have direct access to the facebook/react repo on GitHub. So, if I wanted to contribute to React, I would "fork", or make a copy, of the React Project so I could work on my own version and make improvements. My fork could then be found at asg017/react. Once I am done making my changes, if I wanted my changes to be in the original projects, I would make a "Pull Request."

A Pull Request is done when you want the changes you made to your local forked repo to appear on the original repository. The Pull Request is sent to the people in charge of the original repository (i.e. Facebook), And they will review the changes and decide if they want them in their project. If they approve, then your changes will be "merged" into that repository and you will be an official contributor!

Step 5: Commit your Changes, make a Pull Request

We are now done with all the code changes that we need to include your slash command into our Slack App!

However, our changes are only local, meaning that the new repository that we forked has no idea that we made changes. So we are going to need to add, commit, and push our changes to our forked repository.

And then, those changes that we push will only be shown in the forked repo that you own. ECE USC's repo will not know about these changes. So, we will make a Pull Request (PR) on GitHub to take the changes made in your repo and put them on ECE USC's repo.

As soon as the PR is created, the owners of the ECE USC repo (me) will be able to review the changes and merge it to our product!

  1. Commit your changes to your profile's forked repository.

    2. To do these, we will run a series of git commands in the terminal.

    1. Add your files to the "staging area" with git add --all
      2. What does add do?

      git add tells git "I wanna include these files into my next commit,", which basically means "I want to change these files." The --all flag is a shortcut that'll add every single changed file to the "staging area." If you want to add files one at a time, you could use git add example.py.

    2. "Commit" to your changes with git commit -m "adding to the server"
      3. What does commit do?

      git commit tells git "commit all the changes I've made that's in my staging area", which basically means "go ahead and record the changes I've made and git add'ed already". The -m flag allows you to pass a message with a short description as a summary of the changes you made for that commit. Try to keep it short and descriptive. Good examples are:

      • made token management better
      • updated to make heroku friendly
      • updated requirements, added requests
      Bad ones:
      • whoops

      As you can see from the links above, once you do the next step and push, you can see a webpage on GitHub of each single commit you make, along with the message, what files you changed, and everything else. Pretty useful!

    3. Push your changes to your repository with git push origin master
      4. What does push do?

      git push tells git "push my changes outside of my local computer and share it with the world!". git push takes 2 parameters - a remote_name and branch_name.

      For most of what you do, remote_name will always be origin, which is the default name that GitHub uses. Git remotes are used in more complex project, we will only need to deal with 1 remote.

      branch_name will almost always be master as well. Git branches are used if you want different versions of your code - for example, a single git repo may have a master branch where all the official code is, an experiment branch where developers try experimental features, and a integration branch where they test new features before officially adding to master. For now, we will only work with 1 branch - master, in all of our work.

    All in all, this will be the code you run everytime you want to put your local changes into your repo: 

    
git add --all
git commit -m "Your short, descriptive message here"
git push origin master

    If you head to the GitHub repo of your fork, you will notice your changes!
